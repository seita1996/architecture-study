## この回で目指す状態

今日のゴールは、用語を暗記することではない。

次のように言える状態を目指す。

- これはどのスコープの話か
- これはどの設計軸の話か
- これはどの問いに答える設計判断か
- これはライブラリやフレームワークの話か

用語を箱に分類するだけでなく、どの問題を解こうとしているかを分けられると議論がかなり楽になる。

<!--

-->

## なぜ最初に用語をそろえるのか

同じ言葉を使っていても、見ているものが違うことがある。

| 発言 | 実は見ているもの |
|---|---|
| Clean Architecture にしたい | 依存方向を整理したい |
| Repository を作りたい | DB の詳細を隠したい |
| Service が太い | 責務が集まりすぎている |
| Vertical Slice にしたい | 機能単位で変更しやすくしたい |

言葉の正解探しではなく、問題意識をそろえるのが目的。

<!--
話すこと:
- いきなり用語の定義に入らず、現場で起きる困りごとから話す。
- 設計の話は抽象的に見えるが、変更時の迷い、影響範囲、レビューの難しさを減らすためのものだと結びつける。
- 参加者に、最近変更が怖かった箇所を一つ思い出してもらう。
-->

## まずは身近な例

「請求書を発行する」機能を考える。

```ts
type IssueInvoiceInput = {
  contractId: string
  requestedBy: string
}

type IssueInvoiceResult =
  | { type: "issued"; invoiceId: string }
  | { type: "contract_not_found" }
  | { type: "already_issued" }
```

この小さな処理にも、いろいろな設計判断が入る。

## 同じ機能でも、見る角度が違う

| 見る角度 | 問い |
|---|---|
| 業務ルール | いつ発行できるのか |
| データ | 何を保存するのか |
| 外部依存 | DB、メール、Queue をどう扱うのか |
| テスト | どこを差し替えて確認するのか |
| 変更 | 仕様変更時にどこを触るのか |

アーキテクチャの議論は、この複数の問いを同時に扱う。

## アーキテクチャはフォルダ名ではない

例えば、次のフォルダ構成だけを見ても設計は判断できない。

```txt
features/
  issue-invoice/
    handler.ts
    use-case.ts
    repository.ts
```

大事なのは、名前よりも次のこと。

- どちらがどちらに依存しているか
- どこに業務ルールがあるか
- 外部技術の詳細がどこまで漏れているか
- 変更時にどこまで影響するか

<!--
話すこと:
- コード例は文法の細部より、依存の向き、責務の置き場所、変更時に触る範囲を見る。
- TypeScript では type と const 関数を使った表現でも、設計上の境界や契約を表せることを確認する。
- この形を必ず採用するという話ではなく、何を隠し、何を明示しているかを読む。

## 1. 設計原則

設計原則は「迷ったときの判断基準」。

例:

- 高凝集
- 疎結合
- 関心の分離
- SOLID
- Dependency Inversion

原則はコードを直接作ってくれるものではない。

「この依存は逆にした方がよいのでは？」のように、考える方向を示す。

Dependency Inversion は、高水準の方針が低水準の詳細に直接依存しないようにする原則。
高水準と低水準の両方が、業務上意味のある抽象へ依存する。

- 最後に、現在の設計判断にどう関係するかを一言でつなげる。

## 原則の例: 疎結合

疎結合とは「何かを変えたとき、関係ない場所まで壊れにくい」こと。

```ts
type SendInvoiceMail = (invoiceId: string) => Promise<void>

const issueInvoice =
  (deps: { sendMail: SendInvoiceMail }): IssueInvoice =>
  async (input) => {
    const invoice = await saveInvoice(input)
    await deps.sendMail(invoice.id)
    return { type: "issued", invoiceId: invoice.id }
  }
```

メール送信の実装詳細を、請求書発行の中に直接書かない。

## 2. 設計軸

設計判断は、複数の独立した軸に分けると見通しがよくなる。

| 軸 | 例 |
|---|---|
| コード配置 | Package by Layer、Package by Feature、Vertical Slice |
| 依存方向 | Direct Dependency、Ports and Adapters |
| 業務ロジック | Transaction Script、Domain Model |
| プロセス境界 | Monolith、Microservices |
| 通信スタイル | Direct Call、Queue、Event-driven |
| 読み書き | Shared Model、CQRS |

後半の回では、この軸を一つずつ見ていく。

## アーキテクチャスタイルの例

アーキテクチャスタイルでは、主に責務と依存規則を見る。

```txt
Layered:
Presentation -> Application -> Domain
Infrastructure implements what inner layers need
```

```txt
Hexagonal:
Adapter -> Port -> Application / Domain
Adapter implements Port
```

コード配置は別軸で見る。

```txt
Package by Layer:
controllers/ services/ repositories/

Package by Feature / Vertical Slice:
features/issue-invoice/
```

どちらが常に正しい、という話ではない。

## パターンの例: Repository

Repository は、DB の詳細を業務処理から隠したいときに使う。


ただし、すべてのテーブルに必ず Repository を作る、という意味ではない。

## パターンの例: Strategy

Strategy は、ルールを差し替えたいときに使う。

```ts
type PricingStrategy = (order: Order) => Money

const regularPricing: PricingStrategy = (order) => order.total

const campaignPricing: PricingStrategy = (order) =>
  order.total.multiply(0.9)
```

「料金計算」という呼び出し方は同じで、具体的なルールだけを差し替える。

## 技術と設計判断を分ける

同じ Prisma を使っていても、設計は変わる。

| Prisma の使い方 | 設計上の意味 |
|---|---|
| ユースケースから直接呼ぶ | 実装は少ないが DB 詳細が見える |
| Repository の中で使う | DB 詳細を境界の外へ寄せる |
| Mapper でドメイン型に変換する | DB モデルと業務モデルを分ける |

「Prisma を使うか」だけではなく、「どこで使うか」が設計判断になる。
