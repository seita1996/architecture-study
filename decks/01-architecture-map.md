## この回で目指す状態

今日のゴールは、用語を暗記することではない。

次のように言える状態を目指す。

- これはどのスコープの話か
- これはどの設計軸の話か
- これはどの問いに答える設計判断か
- これはライブラリやフレームワークの話か

用語を箱に分類するだけでなく、どの問題を解こうとしているかを分けられると議論がかなり楽になる。
分類は業界で一意に決まるものではない。
この教材では、議論の粒度をそろえるための便宜上の地図として使う。

<!--

## なぜ最初に用語をそろえるのか

同じ言葉を使っていても、見ているものが違うことがある。

| 発言 | 実は見ているもの |
|---|---|
| Clean Architecture にしたい | 依存方向を整理したい |
| Repository を作りたい | DB の詳細を隠したい |
| Service が太い | 責務が集まりすぎている |
| Vertical Slice にしたい | 機能単位で変更しやすくしたい |

言葉の正解探しではなく、問題意識をそろえるのが目的。

## Architecture Driver

Architecture Driver は、設計判断を動かす要求や制約。

例:

- 主要な機能要求
- 品質特性
- 技術的制約
- 組織・納期・スキル上の制約
- 既存システムとの互換性
- 業務不変条件

「Clean Architecture にするか」より先に、「何を守りたいのか」を確認する。

<!--
話すこと:
- Driver は流行や好みではなく、設計判断の理由になるものだと説明する。
- すべての品質を同時に最大化することはできないため、どれを重く見るかを言語化する必要がある。
- 今日以降のパターン名は、この Driver に対する候補として扱う。
-->
---

## Driver から設計への変換

同じ請求書発行でも、Driver が変わると設計候補も変わる。

| 要求・制約 | 設計への影響 |
|---|---|
| 二重発行を絶対に防ぎたい | unique constraint、transaction、冪等性 |
| メールは遅れてもよい | Queue、Outbox |
| 外部APIが不安定 | 外部APIの詳細を境界へ隔離し、失敗時の扱いを明示する |
| 一覧検索が複雑 | 専用 Query、Read Model |
| 変更が発行機能単位で入る | Vertical Slice |
| 1チーム・1デプロイ | Modular Monolith が初期仮説 |
| 監査証跡が必要 | 監査ログ、場合により Event Sourcing を比較 |

<!--
話すこと:
- この表の右側は唯一の正解ではなく、検討候補だと強調する。
- 例えば二重発行はRepositoryを作るだけでは防げない。DB制約、transaction、冪等性のような実行時の保証が必要になる。
- 「1チーム・1デプロイ」は品質特性ではなく組織制約、「二重発行禁止」は業務不変条件だと分ける。
- 「1チーム・1デプロイなら必ずモノリス」ではなく、最初に疑うべき候補として扱う。
- Read Model は、検索や表示に合わせた読み取り用モデルのこと。詳細は第11回で扱う。
-->
---

## 設計判断の基本形

設計判断は、次の順に書くと説明しやすい。

```txt
Driver
  -> 候補となる設計
  -> 得られる品質
  -> 増えるコスト
  -> 判断と見直し条件
```

例:

- Driver: 外部APIが不安定
- 候補: 外部APIの詳細を境界へ隔離する
- 候補: 失敗時の扱いをアプリの結果型として明示する
- 得るもの: 技術詳細と失敗表現の隔離
- 増えるもの: 抽象、失敗処理、運用確認
- 詳細: Retry、Timeout、Backoff、Circuit Breaker は第10回で扱う

<!--
話すこと:
- Port + Adapter は障害そのものを消すのではなく、プロトコルや失敗表現を隔離するものだと補足する。
- 第1回ではRetryやTimeoutの設計に深入りしない。要求から判断を始める型だけを持ち帰る。
- この形式は第12回のADRまで繰り返し使う。
- 参加者には、今日の最後にこの形で1行書いてもらうと予告する。
-->
---

## アーキテクチャはフォルダ名ではない

例えば、次のフォルダ構成だけを見ても設計は判断できない。

```txt
features/
  issue-invoice/
    handler.ts
    use-case.ts
    repository.ts
```

大事なのは、名前よりも次のこと。

- どちらがどちらに依存しているか
- どこに業務ルールがあるか
- 外部技術の詳細がどこまで漏れているか
- 変更時にどこまで影響するか

<!--
話すこと:
- コード例は文法の細部より、依存の向き、責務の置き場所、変更時に触る範囲を見る。
- TypeScript では type と const 関数を使った表現でも、設計上の境界や契約を表せることを確認する。
- この形を必ず採用するという話ではなく、何を隠し、何を明示しているかを読む。
-->

## 1. 設計原則

設計原則は「迷ったときの判断基準」。

例:

- 高凝集
- 疎結合
- 関心の分離
- SOLID
- Dependency Inversion

原則はコードを直接作ってくれるものではない。

「この依存は逆にした方がよいのでは？」のように、考える方向を示す。

Dependency Inversion は、高水準の方針が低水準の詳細に直接依存しないようにする原則。
高水準と低水準の両方が、業務上意味のある抽象へ依存する。

## 原則の例: 疎結合

疎結合とは「何かを変えたとき、関係ない場所まで壊れにくい」こと。

```ts
type SendInvoiceMail = (invoiceId: string) => Promise<void>

const issueInvoice =
  (deps: { sendMail: SendInvoiceMail }): IssueInvoice =>
  async (input) => {
    const invoice = await saveInvoice(input)
    await deps.sendMail(invoice.id)
    return { type: "issued", invoiceId: invoice.id }
  }
```

メール送信の実装詳細を、請求書発行の中に直接書かない。

## 2. 設計軸

設計判断は、複数の独立した軸に分けると見通しがよくなる。

| 軸 | 例 |
|---|---|
| 変更単位・コード配置 | Package by Layer、Package by Feature、Vertical Slice |
| 依存境界 | Direct Dependency、Ports and Adapters |
| 業務ロジック | Transaction Script、Domain Model |
| データアクセス | ORM直接利用、Mapper、Repository |
| 一貫性 | Transaction、Unique constraint、Idempotency |
| メッセージ意味 | Command、Event |
| 応答 | Sync、Async |
| 配信形態 | Point-to-point、Publish-subscribe |
| 実装 | Function、HTTP、Queue、Broker |
| デプロイ境界 | Monolith、Microservices |
| 読み書き | Shared Model、CQRS |

後半の回では、この軸を一つずつ見ていく。

## アーキテクチャスタイルの例

アーキテクチャスタイルでは、主に責務と依存規則を見る。

一般的なレイヤード:

```txt
Presentation -> Application / Service -> Data Access -> Database
```

依存を逆転したレイヤード:

```txt
Presentation -> Application -> Domain
Infrastructure -> Application / Domain contracts
```

後者は Dependency Inversion を適用した構成で、Clean / Hexagonal に近い考え方も含む。

```txt
Hexagonal:
Adapter -> Port -> Application / Domain
Adapter implements Port
```

コード配置は別軸で見る。

```txt
Package by Layer:
controllers/ services/ repositories/

Package by Feature / Vertical Slice:
features/issue-invoice/
```

どちらが常に正しい、という話ではない。

## 技術と設計判断を分ける

同じ Prisma を使っていても、設計は変わる。

| Prisma の使い方 | 設計上の意味 |
|---|---|
| ユースケースから直接呼ぶ | 実装は少ないが DB 詳細が見える |
| Repository の中で使う | DB 詳細を境界の外へ寄せる |
| Mapper でドメイン型に変換する | DB モデルと業務モデルを分ける |

「Prisma を使うか」だけではなく、「どこで使うか」が設計判断になる。

<!--
話すこと:
