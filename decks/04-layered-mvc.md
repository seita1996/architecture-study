## この回で目指す状態

Layered Architecture を「古い」「普通」「悪い」と雑に扱わない。

次のように説明できる状態を目指す。

- Layer は論理的な責務分割である
- Tier は実行環境やプロセスの分割である
- MVC と Layered は同じ話ではない
- Controller、Service、Repository の分割が効く場面と苦しくなる場面を言える

## なぜ最初に Layered を扱うのか

多くの人が最初に触れる構成だから。

そして、他のアーキテクチャを理解するときの比較対象になる。

| 比較したいもの | Layered を知ると見えること |
|---|---|
| Clean Architecture | 依存方向をより厳格にする理由 |
| Hexagonal | 外部技術を Adapter に寄せる理由 |
| Vertical Slice | 技術レイヤーではなく機能でまとめる理由 |
| Transaction Script | Service に処理を書くときの自然さ |

## まず Layer とは何か

Layer は「役割ごとの段」。

代表例:

- HTTP を受ける
- アプリケーション処理を進める
- 業務ルールを表現する
- DB や外部 API と話す

役割ごとに置き場所を決めると、初見で探しやすくなる。

## 論理レイヤーと物理 tier

| 用語 | 意味 |
|---|---|
| Layer | コード上の責務分割 |
| Tier | 実行環境、プロセス、ネットワーク境界 |
| MVC | UI 入力、表示、状態の分担 |
| Service Layer | アプリケーション操作の入口 |

Layered Architecture と N-tier は同じ言葉ではない。

## 個人ワーク: 変更はどこに散らばるか

Layered な構成で「請求書発行に承認フローを追加する」場合を考える。

```txt
controllers/invoice-controller.ts
services/invoice-service.ts
repositories/invoice-repository.ts
models/invoice.ts
tests/invoice-service.test.ts
```

追加仕様:

- 10万円以上の請求書は `pending_approval` にする
- 承認済みになったら発行メールを送る
- 一覧画面で承認待ちを絞り込む

考えること:

- どのファイルを触りそうか
- どの変更は業務ルールか
- どの変更は表示や永続化の都合か

## 答え合わせ: Layered では技術責務ごとに散る

触る場所はおおよそこうなる。

| 変更 | 触る場所 |
|---|---|
| 入力値を受ける | Controller |
| 10万円以上の判定 | Service または Domain |
| 状態を保存する | Repository / Model |
| 承認待ちの検索 | Repository / Controller |
| メール送信 | Service から外部処理を呼ぶ |
| テスト | Service と Repository 周辺 |

Layered は責務が分かりやすい一方、1つの機能変更が複数ディレクトリへ広がりやすい。

## 今日の判断基準

Layered は悪い構成ではない。

ただし、機能単位で変更されるシステムでは、変更対象が技術レイヤーに散らばるコストを意識する。
