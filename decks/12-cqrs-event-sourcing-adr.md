## なぜ最後に扱うのか

CQRS や Event Sourcing は、強力だが重い。

基礎を飛ばして使うと、次のようになりやすい。

| 誤解 | 実際 |
|---|---|
| CQRS は Command と Query を分けるだけ | 読み取りモデルと書き込みモデルを分ける設計判断 |
| Event Sourcing はログを残すこと | 状態の正本をイベント履歴にすること |
| 先進的だから使う | 要件に合うときだけ使う |
| Clean と組み合わせればよい | 目的が違う |

<!--
話すこと:
- いきなり用語の定義に入らず、現場で起きる困りごとから話す。

- 参加者に、最近変更が怖かった箇所を一つ思い出してもらう。

## まず読み書きを分けて考える

多くの画面では、読み取りと書き込みの要求が違う。

| 操作 | 欲しいもの |
|---|---|
| 請求書を発行する | 正しい業務ルール、整合性 |
| 請求書一覧を見る | 速い検索、並び替え、絞り込み |
| 月次レポートを見る | 集計済みデータ |
| 監査履歴を見る | 過去の変更履歴 |

読み書きの違いが大きいほど、分ける理由が出てくる。

## CQS と CQRS

| 用語 | 意味 |
|---|---|
| CQS | メソッドを Command と Query に分ける原則 |
| CQRS | 書き込みモデルと読み取りモデルを分離するアーキテクチャパターン |

```txt
Command -> Write Model -> Database
Query   -> Read Model  -> Database
```

CQRS は、読み取りと書き込みで異なるモデルを用いる設計。
コード、モデル、スキーマ、保存先をどこまで分離するかは要求に応じて決める。

同じ DB を使う CQRS も、別 DB を使う CQRS もありうる。

## Event Sourcing

現在状態ではなく、状態変化の履歴を保存する。

```txt
InvoiceCreated
InvoiceItemAdded
InvoiceIssued
InvoicePaid
```

CQRS と Event Sourcing は別のパターンであり、必ず組み合わせる必要はない。

<!--
話すこと:

- TypeScript では type と const 関数を使った表現でも、設計上の境界や契約を表せることを確認する。
- この形を必ず採用するという話ではなく、何を隠し、何を明示しているかを読む。
-->

## Event Sourcing をゆっくり見る

通常は「現在の状態」を保存する。

Event Sourcing は「状態を変えた出来事」を保存する。

| 通常の保存 | Event Sourcing |
|---|---|
| invoice.status = paid | InvoicePaid |
| 最新状態が正本 | イベント履歴が正本 |
| 読みやすい | モデル化された状態変化を再現できる |
| 履歴は別途必要 | 現在状態を再構築する必要がある |

<!--

- 一つを優秀、一つを劣っていると扱わず、問題設定が変わると選択も変わると説明する。
- 参加者には、現在のプロダクトならどの列が重要かを考えてもらう。

## 最終比較は「案」ではなく「軸」で見る

次の組み合わせはすべて成立する。

- Vertical Slice + Transaction Script
- Layered + Hexagonal
- Vertical Slice + Hexagonal + CQRS
- Modular Monolith + Event-driven
- Layered + Domain Model + CQRS

Layered、Hexagonal、CQRS、Event-driven は、同じ問いへの排他的な答えではない。

<!--
話すこと:
- ここで「高度な構成へ進化する一本道」という誤解を止める。
- どの軸の話をしているかを明示しないと、Layered と CQRS のような別軸を比較してしまう。
- 以降の ADR は、この表の各軸に対する判断を記録するものだと説明する。
-->

## 設計軸のチェックリスト

| 軸 | 選択肢 |
|---|---|
| コード配置 | Package by Layer / Package by Feature / Vertical Slice |
| 依存境界 | Direct Dependency / Ports and Adapters |
| 業務ロジック | Transaction Script / Domain Model |
| 永続化 | Prisma 直接利用 / Repository / Unit of Work |
| 読み書き | Shared Model / CQRS |
| 連携 | Direct Call / Queue / Event-driven |
| デプロイ | Monolith / Modular Monolith / Microservices |

現在の構成は、各軸で個別に選ぶ。

<!--
話すこと:
- 縦に一つ選ぶ表ではなく、横に軸を並べて一つずつ判断する表だと強調する。
- Event Sourcing は状態保存の戦略であり、CQRS や Event-driven と必ずセットではないことも補足する。
- チームで ADR を書くとき、この軸を見出しにすると議論が散らばりにくい。

## ADR 草案

```md
# ADR: Vertical Slice を基本にし、必要な Slice だけ Ports and Adapters を使う

## Decision
コード配置は Vertical Slice を基本にする。
外部 I/O が複雑な Slice では Ports and Adapters を使う。

## Context
- 請求、契約、通知の変更が機能単位で入ることが多い
- DB、メール、Queue の詳細がユースケースに漏れるとテストが重くなる
- 単純な参照系や管理画面 CRUD もあり、全機能に厚い境界を強制したくない

## Reason
機能単位の凝集度を高める。
外部 I/O が変わりやすい箇所だけ、境界で分離する。

## Trade-offs
Slice ごとの構成判断が必要になる。
Port を作る基準をレビューでそろえる必要がある。

## Rules
- 単純 CRUD は Prisma 直接利用を許容する
- 外部 API、Queue、複雑な DB mapping は Port 化を検討する
- Slice 間の直接依存は避け、共有する場合は理由を残す
- CQRS は読み取り要件が強く分かれるまで採用しない
```

<!--
話すこと:
- Context は「よくあるバックエンドだから」では弱い。どの変更や失敗を受けて決めたのかを書く必要がある。
- Rules に例外と基準を書くと、後日のレビューで「なんとなく Port を作る」を避けやすい。
- ADR は採用パターンの宣言ではなく、制約と判断の記録だと伝える。
-->
---

## 個人ワーク: 軸ごとに選ぶ

「請求書を発行し、メール送信し、監査ログを残す」ユースケースを考える。

制約:

- 発行処理は画面から同期的に実行する
- メール送信は失敗しても再試行できればよい
- 監査ログは必ず残したい
- 読み取り画面は今のところ単純な一覧だけ

各軸で選ぶ:

- コード配置
- 依存境界
- 業務ロジック
- 永続化
- 連携
- 読み書き

すべてを一つのパターンセットとして選ばない。

- 正解を急がず、どの軸で迷ったかを自分で印を付けるくらいで十分だと伝える。

## 答え合わせ: 必要なトレードオフだけ選ぶ

判断例:

| 軸 | 判断例 |
|---|---|
| コード配置 | 請求書発行 Slice にまとめる |
| 依存境界 | メール送信は Port + Adapter にする |
| 業務ロジック | 単純なら Transaction Script、状態遷移が濃ければ Domain Model |
| 永続化 | DB 更新と監査ログは同一ローカルトランザクション |
| 連携 | メール送信だけ Outbox + Queue を検討する |
| 読み書き | 一覧が単純なので CQRS は採用しない |

ADR にするとこう書ける。

```txt
Decision:
  請求書発行は Slice としてまとめる。
  メール送信は Outbox 経由にする。
  CQRS は採用しない。
Context:
  メールは再試行できればよいが、監査ログは発行処理と一緒に残したい。
Reason:
  同期処理の整合性と、非同期通知の再試行性を分けたい。
Trade-offs:
  Outbox の運用と再送監視が増える。
```

選ばなかった軸まで書くと、流行ではなく制約から説明できる。

<!--
話すこと:
- 答え合わせでは唯一の正解としてではなく、判断の筋道として説明する。
- 監査ログが必須なら DB 更新と同じトランザクションに寄せる、メールは再試行でよいなら Outbox を検討する、という前提の使い方を示す。
- 最後に、この回の判断基準へ短く接続する。
-->
---

## まとめ: ADR で設計判断を残す

最終的に残したいのは、採用したパターン名だけではない。

| 残すこと | 例 |
|---|---|
| Decision | どの軸で何を選ぶか |
| Context | どの変更、障害、制約が問題だったか |
| Reason | なぜその軸でその選択をしたか |
| Trade-offs | 増える実装量、学習コスト、運用コスト |
| Rules | Port を作る基準、例外、見直し条件 |
| Rejected Options | なぜ CQRS や Microservices を選ばないか |

設計判断は、選んだ理由と選ばなかった理由をセットで残す。

- ADR は立派な文書を書くためではなく、後から判断理由を追えるようにするためのものだと説明する。
