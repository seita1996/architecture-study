## なぜ最後に扱うのか

CQRS や Event Sourcing は、強力だが重い。

基礎を飛ばして使うと、次のようになりやすい。

| 誤解 | 実際 |
|---|---|
| CQRS は Command と Query を分けるだけ | モデルや保存先を分ける設計判断 |
| Event Sourcing はログを残すこと | 状態の正本をイベント履歴にすること |
| 先進的だから使う | 要件に合うときだけ使う |
| Clean と組み合わせればよい | 目的が違う |

## まず読み書きを分けて考える

多くの画面では、読み取りと書き込みの要求が違う。

| 操作 | 欲しいもの |
|---|---|
| 請求書を発行する | 正しい業務ルール、整合性 |
| 請求書一覧を見る | 速い検索、並び替え、絞り込み |
| 月次レポートを見る | 集計済みデータ |
| 監査履歴を見る | 過去の変更履歴 |

読み書きの違いが大きいほど、分ける理由が出てくる。

## CQS と CQRS

| 用語 | 意味 |
|---|---|
| CQS | メソッドを Command と Query に分ける原則 |
| CQRS | 書き込みモデルと読み取りモデルを分離するアーキテクチャパターン |

```txt
Command -> Write Model -> Database
Query   -> Read Model  -> Database
```

## Event Sourcing

現在状態ではなく、状態変化の履歴を保存する。

```txt
InvoiceCreated
InvoiceItemAdded
InvoiceIssued
InvoicePaid
```

CQRS と Event Sourcing は別のパターンであり、必ず組み合わせる必要はない。

## Event Sourcing をゆっくり見る

通常は「現在の状態」を保存する。

Event Sourcing は「状態を変えた出来事」を保存する。

| 通常の保存 | Event Sourcing |
|---|---|
| invoice.status = paid | InvoicePaid |
| 最新状態が正本 | イベント履歴が正本 |
| 読みやすい | 履歴を完全に追える |
| 履歴は別途必要 | 現在状態を再構築する必要がある |

## 案 A: Layered + Transaction Script

適する状況:

- 単純な CRUD
- 小規模

主なコスト:

- 処理が複雑化すると Service が肥大化しやすい
- 機能変更が複数レイヤーに散らばる

## 案 C: CQRS + Event-driven

適する状況:

- 読み書きの要件が大きく異なる

- 履歴や監査が重要

主なコスト:

- モデルと同期処理が増える
- 実行時の複雑性が高い

## 最終比較

| 観点 | A: Layered | B: Vertical + Hexagonal | C: CQRS + Event-driven |
|---|---|---|---|
| 実装量 | 少ない | 中 | 多い |
| 学習コスト | 低い | 中 | 高い |
| 変更容易性 | 小規模で高い | 機能単位で高い | 要件が合えば高い |
| テスト容易性 | Service 中心 | UseCase 中心 | 契約と非同期処理が重要 |
| 実行時の複雑性 | 低い | 中 | 高い |
| 障害解析 | 容易 | 中 | 難しい |
| 運用コスト | 低い | 中 | 高い |

## ADR 草案

```md
# ADR: Vertical Slice Architecture と Hexagonal Architecture を組み合わせる

## Decision
Vertical Slice Architecture と Hexagonal Architecture を組み合わせる。

## Context
機能単位の変更が多く、DB、外部 API、Queue などの依存がある。

## Reason
機能単位の凝集度を高めつつ、外部技術への依存を境界で分離したい。

## Trade-offs
ファイル数と抽象化が増える。単純な CRUD には過剰になる可能性がある。
```

## 個人ワーク: どの設計を選ぶか

「請求書を発行し、メール送信し、監査ログを残す」ユースケースを考える。

制約:

- 発行処理は画面から同期的に実行する
- メール送信は失敗しても再試行できればよい
- 監査ログは必ず残したい
- 読み取り画面は今のところ単純な一覧だけ

候補:

- 案 A: Layered + Transaction Script
- 案 B: Vertical Slice + Hexagonal
- 案 C: CQRS + Event-driven

どれを選び、どれを選ばないか考える。

## 答え合わせ: 今回の制約なら B が説明しやすい

判断例:

| 案 | 判断 |
|---|---|
| A | 単純なら速いが、メールや監査ログの外部依存が増えると混ざりやすい |
| B | ユースケース単位に集めつつ、メールやDBを Adapter に分けられる |
| C | 読み取り要件が単純なので、CQRS まで入れる理由はまだ弱い |

ADR にするとこう書ける。

```txt
Decision: Vertical Slice + Hexagonal を採用する
Context: 機能単位の変更が多く、DB と外部通知の依存がある
Reason: ユースケースの凝集度を保ち、外部依存を境界で扱いたい
Trade-offs: ファイル数と抽象化が増え、単純 CRUD には過剰になる
```

選ばなかった理由まで書くと、流行ではなく制約から説明できる。

## まとめ: ADR で設計判断を残す

最終的に残したいのは、採用したパターン名だけではない。

| 残すこと | 例 |
|---|---|
| Decision | Vertical Slice + Hexagonal を採用する |
| Context | 機能単位の変更が多く、外部依存がある |
| Reason | 凝集度を保ち、外部技術への依存を境界で扱いたい |
| Trade-offs | ファイル数と抽象化が増える |
| Rejected Options | CQRS は読み取り要件が単純なので今回は選ばない |

設計判断は、選んだ理由と選ばなかった理由をセットで残す。
