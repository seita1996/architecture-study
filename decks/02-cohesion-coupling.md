## まず言葉をゆっくり分ける

凝集度は、モジュール内部の要素がどれくらい一つの目的に関連しているか。

請求書の状態、発行条件、取消条件が近くにあるなら、請求書という関心でまとまっている。

結合度は、モジュール同士がどれくらい相互依存しているか。

請求書の仕様を変えたら契約やサブスクリプションまで壊れるなら、結合が強すぎる可能性がある。

## 凝集度と結合度

| 観点 | 問い |
|---|---|
| 凝集度 | 一つの目的に関係するものが近くにあるか |
| 結合度 | 他のモジュールにどれくらい依存しているか |
| 変更理由 | 同じ理由で変更されるものが近くにあるか |
| 変更波及 | ある変更がどこまで広がるか |
| 関心の分離 | 異なる関心が同じ場所に混ざっていないか |
| 安定依存 | 変わりにくいものへ依存しているか |

## 変更理由を見る

| コードの形 | 変更理由 | 判断 |

| 似ている | 同じ | 共通化してよい候補 |
| 似ている | 違う | まだ分けておく候補 |
| 違う | 同じ | 近づける候補 |
| 違う | 違う | 分かれていてよい |

## 危険な共通化

```ts
type WorkflowStatus =
  | "draft"
  | "active"
  | "issued"
  | "paid"
  | "expired"
  | "cancelled"

type StatusTransition = {
  from: WorkflowStatus
  to: WorkflowStatus
}

export const canTransition = (
  target: "invoice" | "contract" | "subscription",
  transition: StatusTransition,
): boolean => {
  // すべての状態遷移を共通エンジンに集めた
}
```

共通の状態型と遷移エンジンを作ると、重複は減る。
しかし、請求書、契約、サブスクリプションの変更理由が 1 つの型と関数に集まる。

## DRY と疎結合の衝突

| 変更 | 影響 |
|---|---|
| 請求書に `voided` を追加 | 共通関数を変更 |
| 契約の状態遷移を厳格化 | 共通関数を変更 |
| サブスクリプションだけ表示名が必要 | 共通関数に分岐が増える |

## Connascence の入口

Connascence は、複数箇所が一緒に変わる必要性の強さを見るための言葉。

今回は 1 つだけ見る。

```ts
transfer(fromAccountId, toAccountId, amount)
```

この関数は、呼び出し側が引数の順序を正しく覚えている必要がある。
これを「位置の Connascence」として見る。

```ts
transfer({ fromAccountId, toAccountId, amount })
```

名前付きの object にすると、順序への依存を弱められる。

## 個人ワーク: これは良い共通化か

次のコードを見て、共通の状態遷移エンジンにしてよいか考える。

```ts
type InvoiceStatus = "draft" | "issued" | "paid" | "cancelled"
type ContractStatus = "draft" | "active" | "cancelled"

type WorkflowStatus = InvoiceStatus | ContractStatus

type Transition = {
  from: WorkflowStatus
  to: WorkflowStatus
}

const canTransition = (
  target: "invoice" | "contract",
  transition: Transition,
): boolean => {
  if (target === "invoice" && transition.from === "issued") {
    return transition.to === "paid" || transition.to === "cancelled"
  }

  if (target === "contract" && transition.from === "active") {
    return transition.to === "cancelled"
  }

  return false
}
```

考えること:

- 見た目の重複はどこか
- 仕様変更の理由は同じか
- `cancelled` の意味は両方で同じか
- 片方だけ状態が増えたら、どこに影響するか

次の形式で短く書く。

```txt
Decision:
共通化する / 分ける / 条件付き

Drivers:
重視した変更理由や制約

Trade-offs:
得るものと失うもの
```

## 答え合わせ: 重複と同じ変更理由は別

この例では、見た目だけで共通化を急がない方がよい。

| 観点 | 判断 |
|---|---|
| 見た目 | どちらも status を検証していて似ている |
| 変更理由 | 請求書と契約では状態遷移の理由が違う |
| 言葉の意味 | `cancelled` でも業務上の意味が違う可能性がある |
| 変更波及 | 契約だけに `expired` が増えても請求書には関係ない |

別々の業務概念なら、遷移ルールも分ける。

```ts
const canTransitionInvoice = (
  from: InvoiceStatus,
  to: InvoiceStatus,
): boolean => {
  if (from === "issued") return to === "paid" || to === "cancelled"
  return false
}

const canTransitionContract = (
  from: ContractStatus,
  to: ContractStatus,
): boolean => {
  if (from === "active") return to === "cancelled"
  return false
}
```

ただし、共通のワークフロー基盤がプロダクト上の概念として存在するなら共有エンジンは妥当。
遷移規則の管理、監査、可視化を本当に共通化したいかで判断する。

見た目が同じだから共有するのではなく、共通の変更責任が本当に存在するかを見る。

## 今日の判断基準

- 共通化する前に「同じ変更理由か」を確認する
- 変更の波及範囲を説明できる単位で分ける
- 重複削除より、誤った結合の回避を優先する場面がある
- 共通基盤が実在するなら、重複削除ではなく概念として共有する
