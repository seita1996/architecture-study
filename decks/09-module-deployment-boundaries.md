# 第9回: モジュール境界とデプロイ境界

Modular Monolith と Microservices を、独立性と運用コストから判断する

<!--

- 分散モノリスがなぜ危険か説明できる
- まずモジュラーモノリスで済む状況を判断できる

## 境界には段階がある

| 境界 | 分かれるもの | 主なコスト |
|---|---|---|
| フォルダ | コードの置き場所 | 命名規約 |
| モジュール | import 依存 | 境界ルール |
| プロセス | 実行単位 | 起動、監視 |
| サービス | デプロイ、データ所有権 | ネットワーク、契約、運用 |
| チーム | 開発責任 | 調整、組織設計 |

境界が強くなるほど、独立性を上げられる可能性はある。
ただし、DB、契約、リリースが一緒なら独立性は上がらない。

<!--
話すこと:
- 「可能性」と言う。プロセスを分けただけでは独立しない。
- 同時リリースが必要、DBを共有、同期呼び出しが連鎖するなら分散モノリスになりやすい。
-->
---

## Modular Monolith

- デプロイ単位は 1 つ
- 内部は明確なモジュール境界を持つ
- DB を共有する場合でも、テーブルや書き込み責任を決める
- ネットワーク越しの分散問題を避けられる

```txt
one deployable application
  billing module
  contract module
  user module
  notification module
```

モノリスは「何も分かれていない」という意味ではない。

<!--
話すこと:
- ジュニアには、モノリスという言葉が悪口ではないと明示する。
- 重要なのは内部境界と所有権があるか。
-->
---

## Microservices

マイクロサービスは、かなり多くのものを分ける。

- 独立デプロイ
- データ所有権
- API 契約
- 監視と障害対応
- リリース責任
- チーム責任

```txt
Contract Service -> Invoice Service -> Payment Service
```

サービスを分けると、コード以外の問題が増える。

<!--
話すこと:
- 「小さいサービス」ではなく「独立して運用できるサービス」として扱う。
- データ所有権が曖昧なままサービス化すると、結合だけが遠くなる。
-->
---

## 分散モノリス

サービスは分かれているが、実態としては強く結合している状態。

- 同時リリースが必要
- DB を直接共有している
- 同期 API が連鎖する
- スキーマ変更が同時に必要
- 障害が広く波及する

分けたのに独立して変更、リリース、運用できないなら危険。

<!--
話すこと:
- ここは強調する。分散モノリスは「サービス数が多い」ことではなく、独立していない分散構成のこと。
- 何が結合しているか、契約、データ、時間、リリースの4つで見る。
-->
---

## 個人ワーク: 分ける価値はあるか

請求書機能を3サービスに分ける案を考える。

```txt
Contract Service  契約を管理する
Invoice Service   請求書を発行する
Payment Service   支払い状態を管理する
```

考えること:

- どのサービスがどのデータを持つか
- 契約状態を Invoice Service はどう知るか
- API 契約変更時に同時リリースが必要か
- 障害調査でどのログとトレースを見るか

<!--
話すこと:
- 発言を求めず、各自で「独立できていない箇所」を探してもらう。
- 次の答え合わせでは、マイクロサービスを否定するのではなく、必要条件を整理する。
-->
---

## 答え合わせ: 分けるなら所有権も分ける

| 観点 | 見ること |
|---|---|
| データ所有権 | 他サービスのDBを直接読まない |
| API契約 | 互換性とバージョンを管理できる |
| リリース | 片方だけリリースできる |
| 障害解析 | trace id で流れを追える |
| 開発体験 | ローカル依存が増えすぎない |

独立デプロイや組織境界が必要でないなら、まず Modular Monolith で内部境界を作る選択もある。

<!--
話すこと:
- 「分けない」も設計判断。運用コストを払う理由があるかを見せる。
- 第10回では、サービスを分ける前後に関係なく必要になるメッセージングと失敗設計へ進む。
-->
---

## 今日の判断基準

| 状況 | 判断 |
|---|---|
| 1チームで同時リリースしている | Modular Monolith で境界を強める |
| データ所有権とリリース責任を分けたい | Microservices を検討する |
| DB共有や同時リリースが残る | 分散モノリスに注意する |
| 境界を越える呼び出しが多い | 契約と観測を先に設計する |

次回は、同期・非同期、Command / Event、Outbox など失敗時の設計を扱う。
