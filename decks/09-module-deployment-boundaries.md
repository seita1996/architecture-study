# 第9回: モジュール境界とデプロイ境界

Modular Monolith と Microservices を、独立性と運用コストから判断する

<!--

## この回で目指す状態

今日のゴール:

- モジュール境界とデプロイ境界を区別できる
- Modular Monolith と Microservices の違いを説明できる
- 分散モノリスがなぜ危険か説明できる
- まずモジュラーモノリスで済む状況を判断できる

## 境界には段階がある

| 境界 | 分かれるもの | 主なコスト |
|---|---|---|
| フォルダ | コードの置き場所 | 命名規約 |
| モジュール | import 依存 | 境界ルール |
| プロセス | 実行単位 | 起動、監視 |
| サービス | デプロイ、データ所有権 | ネットワーク、契約、運用 |
| チーム | 開発責任 | 調整、組織設計 |

境界が強くなるほど、独立性を上げられる可能性はある。
ただし、DB、契約、リリースが一緒なら独立性は上がらない。

<!--
話すこと:
- 「可能性」と言う。プロセスを分けただけでは独立しない。
- 同時リリースが必要、DBを共有、同期呼び出しが連鎖するなら分散モノリスになりやすい。
-->
---

## Modular Monolith

- デプロイ単位は 1 つ
- 内部は明確なモジュール境界を持つ
- DB を共有する場合でも、テーブルや書き込み責任を決める
- ネットワーク越しの分散問題を避けられる

```txt
one deployable application
  billing module
  contract module
  user module
  notification module
```

モノリスは「何も分かれていない」という意味ではない。

<!--
話すこと:
- ジュニアには、モノリスという言葉が悪口ではないと明示する。
- 重要なのは内部境界と所有権があるか。
-->
---

## Microservices

マイクロサービスは、かなり多くのものを分ける。

- 独立デプロイ
- データ所有権
- API 契約
- 監視と障害対応
- リリース責任
- チーム責任

```txt
Contract Service -> Invoice Service -> Payment Service
```

この図は、Entity 名で分けた初期仮説にすぎない。
「契約」「請求書」「支払い」という名詞があるから、そのままサービスにするとは限らない。

サービスを分けると、コード以外の問題が増える。

<!--
話すこと:
- 「小さいサービス」ではなく「独立して運用できるサービス」として扱う。
- データ所有権が曖昧なままサービス化すると、結合だけが遠くなる。
- Entityごとにサービスを作る発想は危険な出発点になりやすい。境界は業務能力、不変条件、所有権から見る。
-->
---

## 境界を見つける材料

サービス境界は、名前ではなく独立させたいものから考える。

| 観点 | 見ること |
|---|---|
| 業務能力 | 事業上の能力。例: 請求する、支払いを受ける |
| 不変条件 | 常に守る必要がある業務ルールの範囲か |
| トランザクション | 同一 transaction が必要な範囲か |
| データ所有権 | どのモジュールやサービスが書き込み責任を持つか |
| 変更理由 | 同じタイミングで変わるか |
| チーム所有権 | 誰が責任を持って変更、運用するか |
| 非機能要求 | 独立したスケーリング、可用性、規制があるか |
| 通信頻度 | 境界をまたぐ呼び出しが多すぎないか |

<!--
話すこと:
- どれか一つで決まるのではなく、複数の材料を並べて初期仮説を作る。
- 同一トランザクションが頻繁に必要なものを別サービスにすると、整合性設計が急に難しくなる。
-->
---

## 分散モノリス

サービスは分かれているが、実態としては強く結合している状態。

- 同時リリースが必要
- DB を直接共有している
- 同期 API が連鎖する
- スキーマ変更が同時に必要
- 障害が広く波及する

分けたのに独立して変更、リリース、運用できないなら危険。

<!--
話すこと:
- ここは強調する。分散モノリスは「サービス数が多い」ことではなく、独立していない分散構成のこと。
- 何が結合しているか、契約、データ、時間、リリースの4つで見る。
-->
---

## 個人ワーク: 分ける価値はあるか

請求書機能を3サービスに分ける案を、条件ごとに考える。

```txt
Contract Service  契約を管理する
Invoice Service   請求書を発行する
Payment Service   支払い状態を管理する
```

これはあえて疑うための案。
Entity ごとにサービスを作る判断が妥当かを確認する。

条件:

| 条件 | 状況 |
|---|---|
| A | 1チーム、同時リリース、同じ可用性要求 |
| B | PaymentだけPCI対象、別チーム、独立スケール、独立リリース |
| C | 別サービスだが毎回同期呼び出し、同じDBを更新 |

考えること:

- どのサービスがどのデータを持つか
- 契約状態を Invoice Service はどう知るか
- API 契約変更時に同時リリースが必要か
- 障害調査でどのログとトレースを見るか

次の形式で短く書く。

```txt
Decision:
分ける / モジュラーモノリスに留める / 追加調査

Drivers:
独立させたい要求や制約

Trade-offs:
得るものと増える運用コスト
```

<!--
話すこと:
- 各自で「独立できていない箇所」と判断理由を短く書いてもらう。
- 次の答え合わせでは、マイクロサービスを否定するのではなく、必要条件を整理する。
-->

## 答え合わせ: 条件で判断は変わる

| 条件 | 判断例 |
|---|---|
| A | Modular Monolith で内部境界を強める |
| B | Payment を別サービスにする理由がある |
| C | 分散モノリスの危険が高い |

分けるなら、次も分ける必要がある。

| 観点 | 見ること |
|---|---|
| データ所有権 | 他サービスのDBを直接読まない |
| API契約 | 互換性とバージョンを管理できる |
| リリース | 片方だけリリースできる |
| 障害解析 | trace ID で流れを追える |
| 開発体験 | ローカル依存が増えすぎない |

独立デプロイや組織境界が必要でないなら、まず Modular Monolith で内部境界を作る選択もある。
trace ID は、複数サービスをまたぐ1つの処理をログやトレースで追うための識別子。

<!--
話すこと:
- 「分けない」も設計判断。運用コストを払う理由があるかを見せる。
- 第10回では、サービスを分ける前後に関係なく必要になるメッセージングと失敗設計へ進む。
-->

## 今日の判断基準

| 状況 | 判断 |
|---|---|
| 1チームで同時リリースしている | Modular Monolith で境界を強める初期仮説 |
| データ所有権とリリース責任を分けたい | Microservices を検討する |
| 障害分離、規制、負荷特性が強く違う | 1チームでもサービス分割を検討する |
| DB共有や同時リリースが残る | 分散モノリスに注意する |
| 境界を越える呼び出しが多い | 契約と観測を先に設計する |

次回は、同期・非同期、Command / Event、Outbox など失敗時の設計を扱う。
