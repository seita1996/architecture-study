## この回で目指す状態

今日のゴール:

- 設計判断を軸ごとに説明できる
- 採用しなかった選択肢も説明できる
- 見直し条件を ADR に残せる
- 実コードから改善候補を見つけられる

<!--
話すこと:
- 第12回では新しいパターンを増やさない。
- ここまでの語彙を使って、実プロダクトをレビューする。
-->
---

## 持ち込む材料

実コードから、次の例を1つ以上必ず持ち込む。

- 変更が多い Slice
- Prisma が直接見えている Use Case
- 外部 API を呼ぶ Use Case
- 複数 Slice から参照される Domain
- transaction 境界が曖昧な処理
- Queue を使っている処理

架空の請求書だけでは到達度を測れない。
実際の判断に戻る。

<!--
話すこと:
- 実コードを全員で読む必要はない。講師が短い断片にして持ってくる。
- 企業秘密や巨大なコードをスライドに貼らず、判断材料だけ抽出する。
-->
---

## 設計軸レビュー

| 軸 | 記録すること |
|---|---|
| コード配置 | Package by Layer / Feature / Vertical Slice |
| 依存境界 | Direct Dependency / Ports and Adapters |
| 業務ロジック | Transaction Script / Domain Model |
| データアクセス・マッピング | ORM直接 / Data Mapper / Repository |
| 一貫性・変更追跡 | 明示的 transaction / ORM内蔵管理 / Unit of Work |
| メッセージ意味 | Command / Event |
| 配信・実装 | Sync / Async、HTTP / Queue / Broker |
| デプロイ | Monolith / Modular Monolith / Microservices |

Repository と Unit of Work、Queue と Event-driven のように、共存する軸を混ぜない。

<!--
話すこと:
- 前回指摘された非直交な表をここで解消する。
- 一つの列から選ぶのではなく、各軸で判断する。
-->
---

## ADR に残す項目

```md
# ADR: 請求書発行の境界をどう設計するか

## Decision
何を選ぶか

## Context
どの変更、障害、制約が問題だったか

## Reason
なぜその軸でその選択をしたか

## Trade-offs
増える実装量、学習コスト、運用コスト

## Rules
Port を作る基準、例外、見直し条件

## Rejected Options
なぜ選ばないか
```

<!--
話すこと:
- ADR は立派な作文ではない。あとで判断を追えるようにするための記録。
- Context を抽象化しすぎない。具体的な変更頻度、障害、制約を書く。
-->
---

## 個人ワーク: 実コードをレビューする

参加者ごと、またはペアごとに、実コードの一箇所を選ぶ。

```txt
対象:
どの機能、変更、処理か

Drivers:
要求、品質特性、制約

Current design:
現在の依存、データ、transaction、外部I/O

Problems:
どの変更や障害が難しいか

Alternatives:
最低2案

Decision:
どの案を選ぶか

Trade-offs:
何を得て何を失うか

Unknowns:
追加調査が必要なこと

Review condition:
どの条件で見直すか
```

<!--
話すこと:
- 模範解答をすぐ見せるのではなく、書かれた判断を講師またはチームメンバーがレビューする。
- コード全体ではなく、変更が多い箇所、transaction境界が曖昧な箇所、外部I/Oがある箇所などを短く切り出す。
-->
---

## レビューで見る設計軸

| 軸 | 確認すること |
|---|---|
| コード配置 | 変更単位と配置が合っているか |
| 依存境界 | 外部技術や失敗契約をどこで変換しているか |
| 業務ロジック | Transaction Script / Domain Model のどちらが複雑さに合うか |
| データアクセス | ORM直接、Query、Repository、Mapper のどれが必要か |
| 一貫性 | transaction、unique constraint、冪等性が必要か |
| メッセージング | Command / Event、Sync / Async、Queue / Broker を混同していないか |
| 読み書き | CQRS が必要なほどモデルやスケールが違うか |

これは「Vertical Slice + Hexagonal だから正しい」ではない。
各軸で必要なトレードオフだけ選ぶ。

<!--
話すこと:
- 支払い/メールPortの理由を「差し替え」だけにしない。失敗契約の変換も境界を作る理由になる。
- 一覧までRepositoryに通さない判断も説明する。
-->
---

## 参考: ADR 例

```txt
Decision:
  請求書発行は Slice としてまとめる。
  外部メール送信は Port + Adapter で扱う。
  メール送信は Outbox 経由の Command Message にする。
  CQRS は採用しない。

Context:
  メールAPIは一社固定だがタイムアウトが多い。
  監査ログは請求書更新と一緒に必ず残す必要がある。
  一覧画面は単純で、読み取り最適化の要求はまだ薄い。

Reason:
  同期整合性が必要な処理と、再試行可能な外部通知を分けたい。

Trade-offs:
  Outbox worker、再送監視、冪等Handlerが必要になる。

Review:
  一覧検索が複雑化したら Read Model / CQRS を再検討する。
```

<!--
話すこと:
- Review 条件まで書くことが重要。将来の自分たちが見直せる。
-->

## 到達度の評価

各項目を 0〜2 点で見る。

| 項目 | 2点の状態 |
|---|---|
| 問題設定 | 解決したい問題を具体化できる |
| Driver | 要求、品質特性、制約を列挙できる |
| スコープ | コード、モジュール、プロセス、デプロイを区別できる |
| 設計軸 | 異なる軸のパターンを混同しない |
| 代替案 | 最低2案を出せる |
| Trade-off | 利点とコストを両方説明できる |
| Failure mode | 並行実行、部分失敗、再試行を考慮できる |
| 判断 | 制約に基づいて一案を選べる |
| 不確実性 | 足りない情報を言える |
| 見直し条件 | 将来の再判断条件を残せる |

16点以上なら、シニアのレビュー下で局所的な設計判断に参加できる状態。

<!--
話すこと:
- 評価は暗記確認ではなく、初見の実コードに対して判断を言語化できるかを見る。
- 点数は序列化ではなく、次にどの能力を伸ばすかを見つけるために使う。
-->

## 採用しない判断も残す

| 選択肢 | 今回採用しない理由 |
|---|---|
| 全SliceでPorts and Adapters | 単純CRUDには過剰 |
| 全テーブルRepository | Queryや集計まで意図が曖昧になる |
| CQRS | 読み取り要件がまだ単純 |
| Event Sourcing | イベント列を状態の正本にする要求がない |
| Microservices | 独立デプロイやデータ所有権分離の要求がまだ弱い |

採用しなかった理由が残ると、流行ではなく制約から説明できる。

<!--
話すこと:
- 「やらない」ことを残すと、後でなぜ採用していないのかを説明できる。
-->
