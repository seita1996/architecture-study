## この回で目指す状態

今日のゴール:

- 設計判断を軸ごとに説明できる
- 採用しなかった選択肢も説明できる
- 見直し条件を ADR に残せる
- 実コードから改善候補を見つけられる

<!--
話すこと:
- 第12回では新しいパターンを増やさない。
- ここまでの語彙を使って、実プロダクトをレビューする。
-->
---

## 持ち込む材料

可能なら実コードから、次の例を1つずつ持ち込む。

- 変更が多い Slice
- Prisma が直接見えている Use Case
- 外部 API を呼ぶ Use Case
- 複数 Slice から参照される Domain
- transaction 境界が曖昧な処理
- Queue を使っている処理

架空の請求書だけでなく、実際の判断に戻る。

<!--
話すこと:
- 実コードを全員で読む必要はない。講師が短い断片にして持ってくる。
- 企業秘密や巨大なコードをスライドに貼らず、判断材料だけ抽出する。
-->
---

## 設計軸レビュー

| 軸 | 記録すること |
|---|---|
| コード配置 | Package by Layer / Feature / Vertical Slice |
| 依存境界 | Direct Dependency / Ports and Adapters |
| 業務ロジック | Transaction Script / Domain Model |
| データアクセス・マッピング | ORM直接 / Data Mapper / Repository |
| 一貫性・変更追跡 | 明示的 transaction / ORM内蔵管理 / Unit of Work |
| メッセージ意味 | Command / Event |
| 配信・実装 | Sync / Async、HTTP / Queue / Broker |
| デプロイ | Monolith / Modular Monolith / Microservices |

Repository と Unit of Work、Queue と Event-driven のように、共存する軸を混ぜない。

<!--
話すこと:
- 前回指摘された非直交な表をここで解消する。
- 一つの列から選ぶのではなく、各軸で判断する。
-->
---

## ADR に残す項目

```md
# ADR: 請求書発行の境界をどう設計するか

## Decision
何を選ぶか

## Context
どの変更、障害、制約が問題だったか

## Reason
なぜその軸でその選択をしたか

## Trade-offs
増える実装量、学習コスト、運用コスト

## Rules
Port を作る基準、例外、見直し条件

## Rejected Options
なぜ選ばないか
```

<!--
話すこと:
- ADR は立派な作文ではない。あとで判断を追えるようにするための記録。
- Context を抽象化しすぎない。具体的な変更頻度、障害、制約を書く。
-->
---

## 個人ワーク: 請求書発行をレビューする

制約:

- 発行処理は画面から同期実行
- DB 更新と監査ログは必ず一緒に残したい
- メール送信は遅れてもよいが再試行したい
- 一覧画面は今のところ単純
- 外部APIは一社固定だが、タイムアウトが多い

各軸で判断する。

- コード配置
- 依存境界
- 業務ロジック
- データアクセス
- 一貫性
- メッセージング
- 読み書き

<!--
話すこと:
- どれか一つの構成名を選ばせない。
- 外部APIが一社固定でも、失敗契約を変換するためにPortを作る判断があり得る。
-->
---

## 答え合わせ: 判断例

| 軸 | 判断例 |
|---|---|
| コード配置 | 請求書発行 Slice にまとめる |
| 依存境界 | タイムアウトや失敗をアプリの結果型へ変換するため Payment / Mail Port を作る |
| 業務ロジック | 単純なら Transaction Script、状態遷移が濃ければ Domain Model |
| データアクセス | 書き込みは Repository、一覧は Query でよい |
| 一貫性 | 請求書更新と監査ログは同一 transaction |
| メッセージング | メール送信は Outbox + Command Message |
| 読み書き | CQRS はまだ採用しない |

これは「Vertical Slice + Hexagonal だから正しい」ではない。
各軸で必要なトレードオフだけ選ぶ。

<!--
話すこと:
- 支払い/メールPortの理由を「差し替え」だけにしない。失敗契約の変換も境界を作る理由になる。
- 一覧までRepositoryに通さない判断も説明する。
-->
---

## ADR 例

```txt
Decision:
  請求書発行は Slice としてまとめる。
  外部メール送信は Port + Adapter で扱う。
  メール送信は Outbox 経由の Command Message にする。
  CQRS は採用しない。

Context:
  メールAPIは一社固定だがタイムアウトが多い。
  監査ログは請求書更新と一緒に必ず残す必要がある。
  一覧画面は単純で、読み取り最適化の要求はまだ薄い。

Reason:
  同期整合性が必要な処理と、再試行可能な外部通知を分けたい。

Trade-offs:
  Outbox worker、再送監視、冪等Handlerが必要になる。

Review:
  一覧検索が複雑化したら Read Model / CQRS を再検討する。
```

<!--
話すこと:
- Review 条件まで書くことが重要。将来の自分たちが見直せる。
-->

## 採用しない判断も残す

| 選択肢 | 今回採用しない理由 |
|---|---|
| 全SliceでPorts and Adapters | 単純CRUDには過剰 |
| 全テーブルRepository | Queryや集計まで意図が曖昧になる |
| CQRS | 読み取り要件がまだ単純 |
| Event Sourcing | イベント列を状態の正本にする要求がない |
| Microservices | 独立デプロイやデータ所有権分離の要求がまだ弱い |

採用しなかった理由が残ると、流行ではなく制約から説明できる。

<!--
話すこと:
- 「やらない」ことを残すと、後でなぜ採用していないのかを説明できる。
-->
