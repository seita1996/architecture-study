## なぜこの話が必要か

アプリが大きくなると「分けたい」という話が出る。

でも、分け方には段階がある。

| 分け方 | 何が分かれるか |
|---|---|
| フォルダを分ける | コードの置き場所 |
| モジュールを分ける | 依存関係 |
| プロセスを分ける | 実行単位 |
| DB を分ける | データ所有権 |
| チームを分ける | 開発責任 |

マイクロサービスは、かなり多くのものを分ける。

## まず境界とは何か

境界は「ここから先は別の責任」と決める線。

境界が強くなるほど、独立性は上がる。

一方で、連携コストも上がる。

| 境界 | 独立性 | コスト |
|---|---|---|
| 関数 | 低 | 低 |
| モジュール | 中 | 中 |
| プロセス | 高 | 高 |
| ネットワーク越しのサービス | 高 | とても高い |

## モジュラーモノリスの見方

モノリスは「何も分かれていない」という意味ではない。

よいモジュラーモノリスは、内部に境界を持つ。

例:

- billing module
- contract module
- user module
- notification module

デプロイは 1 つでも、依存関係と所有権は分ける。

## マイクロサービスで増えるもの

| 観測 | tracing、metrics、logs |
| 契約管理 | API version、schema |
| 運用 | deploy、rollback、alert |

小さく分けるほど、運用の設計が必要になる。

## イベントで何が変わるか

関数呼び出しは「相手を知っている」。

イベント発行は「起きた事実だけを知らせる」。

| 関数呼び出し | イベント |
|---|---|
| 相手を直接呼ぶ | 起きたことを発行する |
| 処理順が見えやすい | 反応する処理が分散する |
| 失敗がすぐ返る | 非同期だと後から失敗する |
| 追いやすい | 追跡には仕組みが必要 |

## 分散モノリス

- DB を直接共有している
- 同期 API が連鎖する
- 障害が広く波及する
- 契約変更に弱い

## 個人ワーク: サービス分割で何が増えるか

請求書機能を3サービスに分ける案を考える。

```txt
Contract Service  契約を管理する
Invoice Service   請求書を発行する
Payment Service   支払い状態を管理する
```

ユースケース:

```txt
契約が有効なら請求書を発行し、支払い期限を設定する。
支払いが完了したら請求書を paid にする。
```

考えること:

- どのサービスがどのデータを持つか
- 請求書発行中に Payment Service が落ちたらどうなるか
- ローカル開発で何を起動する必要があるか
- 障害調査でどのログを見るか

## 答え合わせ: 境界を増やすと運用の問題も増える

分けることで独立性は増えるが、同時に分散システムの問題が出る。

| 観点 | 難しくなること |
|---|---|
| データ所有権 | 契約状態を Invoice 側で直接読めない |
| 整合性 | 請求書発行と支払い期限作成を1トランザクションにしにくい |
| 障害解析 | 複数サービスのログとトレースを見る必要がある |
| 開発体験 | ローカルで依存サービスが増える |
| リリース | API 契約変更に順序が必要 |

独立デプロイや組織境界が必要でないなら、まずモジュラーモノリスで境界を作る選択もある。

## 今日の判断基準

サービス境界を増やす判断は、コード構成だけでなく運用コストも含めて考える。

| 選択肢 | 見るべき制約 |
|---|---|
| モジュラーモノリス | 1デプロイのまま内部境界を強めたい |
| マイクロサービス | 独立デプロイやデータ所有権が必要 |
| イベント駆動 | 非同期に複数処理を反応させたい |
| 分散モノリス | 分けたのに同時変更が多いなら危険 |
